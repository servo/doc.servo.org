Per ES6, the [[DefineOwnProperty]] internal method has three different possible outcomes:

  • It can throw an exception (which we indicate by returning false).

  • It can return true, indicating unvarnished success.

  • It can return false, indicating โstrict failureโ. The property could not be defined. Itโs an error, but no exception was thrown.

Itโs not just [[DefineOwnProperty]]: all the mutating internal methods have the same three outcomes. (The other affected internal methods are [[Set]], [[Delete]], [[SetPrototypeOf]], and [[PreventExtensions]].)

If you think this design is awful, youโre not alone. But as itโs the standard, we must represent these boolean โsuccessโ values somehow. ObjectOpSuccess is the class for this. Itโs like a bool, but when itโs false it also stores an error code.

Typical usage:

ObjectOpResult result;
if (!DefineProperty(cx, obj, id, ..., result)) {
    return false;
}
if (!result) {
    return result.reportError(cx, obj, id);
}

Users donโt have to call result.report(); another possible ending is:

argv.rval().setBoolean(result.ok());
return true;

Fieldsยง

ยงcode_: usize

code_ is either one of the special codes OkCode or Uninitialized, or an error code. For now the error codes are JS friend API and are defined in js/public/friend/ErrorNumbers.msg.

code_ is uintptr_t (rather than uint32_t) for the convenience of the JITs, which would otherwise have to deal with either padding or stack alignment on 64-bit platforms.
